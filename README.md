## 1️⃣ 아이디어 및 목적 (Why & What)

### 기획 배경: 이 서비스를 왜 만들게 되었나요?: 개발 효율성의 혁신과 '순수한 재미'로의 회귀

[문제점] 복잡함과 진입장벽: 기존 게임 개발은 진입 장벽이 높고, 시중의 게임들은 지나치게 복잡한 과금 모델과 학습 곡선을 요구하여 유저에게 피로감을 줍니다.

[해결책] 바이브 코딩를 통한 초고속 개발: '바이브 코딩' 기술을 도입하여, 물리 엔진과 적의 AI 로직을 갖춘 고퀄리티 게임을 빠르게 구현할 수 있음을 증명했습니다.

[가치] 본질적인 재미 추구: 복잡함을 걷어내고, '직관적인 조작'과 '클리어의 쾌감'이라는 플랫포머 게임의 본질적인 재미를 최신 기술로 재해석했습니다.

### 주요 기능: 사용자에게 제공하는 핵심 기능은 무엇인가요?: 누구나 즐기는 직관적 UX와 단계별 몰입 설계)

1. AI 기반의 정교한 물리 엔진 & 로직: 바이브 코딩으로 구현된 부드러운 캐릭터 움직임과 충돌 처리로 위화감 없는 플레이 경험 제공.

2. 직관적인 'Run & Action' 시스템: 별도의 튜토리얼 없이 방향키와 공격키만으로 즉시 플레이 가능한 초간편 UI/UX.

3. 도전 욕구를 자극하는 '보스 레이드': 단순한 스테이지 진행을 넘어, 고유 패턴을 가진 보스 몬스터와의 전투를 통해 긴장감과 클리어 시 확실한 Reward 제공.

### 타겟 사용자: 이 앱을 누가 사용하면 좋을까요?: 짧은 호흡의 콘텐츠를 선호하는 현대인

- ⚡ 스낵 게이머 (Snack Gamer): 이동 시간이나 휴식 시간에 1~3분 내외로 짧고 굵게 즐길 수 있는 게임을 찾는 학생 및 직장인.

- 🕹️ 뉴트로(New-tro) 매니아: '별의 커비'와 같은 횡스크롤 액션 게임의 향수를 느끼고 싶은 2030 세대.

- 🤖 테크 트렌드 팔로워: AI 코딩 도구(바이브 코딩)가 실제 게임으로 어떻게 구현되었는지 그 결과물과 효용성에 관심이 많은 얼리어답터.

## 3️⃣ AI 협업 및 활용 (AI Collaboration)

### 아이디어 & 구조: AI를 활용해 서비스의 구조나 데이터 모델을 어떻게 잡았나요?

#### 객체 지향 설계와 물리 엔진 구현의 효율화

- 객체 지향(OOP) 구조 설계: 게임의 확장성을 위해 AI에게 '플레이어', '몬스터(잡몹)', '보스'의 공통점과 차이점을 분석 요청하여, 상속 구조(Parent Class: Character -> Child Class: Player, Enemy)를 제안받아 적용했습니다. 이를 통해 코드 중복을 최소화했습니다.

- 물리 엔진 로직 최적화: 플랫포머 게임의 핵심인 '점프'와 '중력' 값을 설정할 때, 바이브 코딩에게 "별의 커비처럼 둥실 떠다니는 느낌(Floaty feeling)"을 요청하여 최적의 가속도 및 마찰력 수치를 데이터 모델로 잡았습니다.


### 구현 & 문제 해결: 코딩 중 막혔던 부분이나 에러를 AI와 어떻게 해결했나요? (사용한 프롬프트 예시 포함)
#### 1. 캐릭터가 바닥을 뚫고 떨어지는 버그 해결 (물리 엔진)

- 상황: 캐릭터가 점프 후 착지할 때, 바닥 타일을 인식하지 못하고 맵 아래로 추락하는 현상 발생.

- 사용 프롬프트:
##### "지금 플레이어 캐릭터가 점프 후 착지할 때 isGrounded 상태가 true로 바뀌지 않고 바닥을 뚫고 지나가. 충돌 박스(Hitbox)의 오차가 있는 것 같은데, 캐릭터의 발 위치와 타일의 상단 좌표를 비교하여 충돌을 감지하는 로직으로 코드를 수정해 줘."

- 결과: AI가 좌표 기반의 충돌 감지 함수를 재작성하여, 착지 판정을 정교하게 개선함.

#### 2. 보스 몬스터의 '상태 머신(State Machine)' 구현 (AI 로직)

- 상황: 보스가 플레이어를 공격하지 않고 가만히 있거나, 패턴이 꼬이는 문제.

- 사용 프롬프트:
##### "보스 몬스터의 행동 패턴을 [대기 - 추적 - 공격 - 피격]의 4가지 상태(State)로 나누고 싶어. 플레이어와의 거리가 50 이하일 때 '공격' 상태로 전환되고, 공격 후에는 2초간 딜레이(대기)를 갖도록 switch 문을 사용해서 상태 관리 로직을 짜줘."

- 결과: 복잡한 if-else 문을 깔끔한 상태 머신 구조로 변경하여 보스의 패턴이 기획 의도대로 작동하게 됨.

#### 3. 스테지이 확장 구현

- 상황: 스테이지가 2개밖에 없어 금방 게임이 끝남.

- 사용 프롬프트:
##### "두가지 밖에 없는 스테이지를 7까지 확장하고 각 스테이지의 테마는 지구의 환경(사막,정글,바다,초원)으로 구성해주고 디디디 대왕이랑 메타 나이트한테 고유 공격 패턴이랑 전용 체력바 넣어서 보스전 만들어주고 보스전은 다른 스테이지와 다르게 보스캐릭터를 죽여야지만 클리어 할 수 있게 만들어줘."

- 결과: 스테이지가 총 7개로 늘어났고, 보스전의 규칙이 새로 생김. 이를 통해 게임이 더 다이나믹해짐.

### 학습 경험: AI와 협업하며 새롭게 알게 된 기능이나 역할에 대해 적어주세요.

#### Syntax(문법)보다 Logic(논리)의 중요성
: 예전에는 오타나 문법 에러를 잡는 데 시간을 썼다면, AI와 협업하면서 "어떤 논리로 코드를 짤 것인가"를 구상하는 설계 능력의 중요성을 깨달았습니다.

#### AI는 완벽한 '페어 프로그래머(Pair Programmer)'
: 내가 모호하게 지시하면 AI도 엉뚱한 코드를 짠다는 것을 알게 되었습니다. 원하는 결과를 얻기 위해 요구사항을 구체적이고 논리적으로 명령하는 '프롬프트 엔지니어링' 역량이 향상되었습니다.

#### 빠른 프로토타이핑
: 바이브 코딩을 통해 아이디어를 즉시 시각적인 결과물로 확인하며 수정하는 '애자일(Agile)'한 개발 방식을 경험했습니다.
